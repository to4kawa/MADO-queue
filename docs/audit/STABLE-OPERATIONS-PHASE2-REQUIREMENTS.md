# MADO-queue 安定運用 Phase 2 要件定義案

## 1. 位置づけ

この文書は、MADO-queue を自治体業務で継続的に利用できる情報システムとして扱うための、安定運用 Phase 2 の要件定義案である。

本稿は upstream への即時提案ではなく、外部レビュー・監査メモとして fork 側に記録する。主に以下の upstream 論点を整理するための作業ノートである。

- Memuro-Town/MADO-queue#14: DB 永続化・バックアップに関する論点
- Memuro-Town/MADO-queue#17: log データの定期削除に関する論点

本稿の目的は、削除機能やバックアップ機能の実装方法を直ちに決めることではない。先に、自治体業務として何を守り、どの記録をどの期間残し、誰がどの範囲で判断するかを整理する。

## 2. 背景

MADO-queue は、Docker Compose により比較的簡単に起動できる構成であり、小規模な業務システムとして導入しやすい利点がある。

一方で、継続運用を前提にすると、以下の論点が発生する。

- DB ファイルをどのように保持・保護するか
- 障害時にどの時点まで復旧できるようにするか
- ログをどの目的で、どの期間保持するか
- 不要になったログをどの条件で削除するか
- 削除前にログや DB を退避する必要があるか
- 非技術者のメンテナーや自治体担当者でも判断・引き継ぎできる形になっているか

特にログ削除や DB 構成変更は、単なる技術的改善ではなく、業務継続、問い合わせ対応、監査、障害解析、説明責任に関係する可能性がある。

## 3. 対象範囲

本要件定義案の対象は以下とする。

- SQLite DB ファイル
- 発券・処理に関するログテーブル
- DB バックアップ
- 障害時復旧
- ログ保持期間
- ログ削除または退避
- 運用手順書
- メンテナーおよび運用担当者の判断範囲

以下は本要件定義案の対象外とする。

- UI の大幅変更
- 業務フローそのものの変更
- 新規機能追加
- 外部クラウドサービスの採用決定
- 本番環境インフラの全面刷新

## 4. 基本方針

### 4.1 先に守る対象を決める

ログや DB を削除・変更する前に、何を業務上の記録として保持すべきかを決める。

削除機能は、保持方針が決まった後に検討する。

### 4.2 非技術者メンテナーが判断できる説明を優先する

技術的に実装可能であっても、メンテナーや自治体担当者が運用上の意味を理解できない変更は、すぐに採用しない。

Pull Request では、少なくとも以下を説明する。

- 現場業務上の利益
- 既存運用への影響
- 障害時の影響
- 戻し方
- 庁内説明が必要かどうか
- メンテナー判断で扱える変更かどうか

### 4.3 軽量な構成を維持する

MADO-queue の価値の一つは、Docker Compose で簡単に起動できる点にある。

外部 DB、外部ログ基盤、クラウドサービスなどを前提にする場合は、導入・保守・費用・責任分界を明示する。

### 4.4 削除より退避を優先する

ログや DB 容量が問題になる場合でも、まずは削除ではなく、バックアップ、退避、保持期間の設定を検討する。

削除は、保持不要であることが確認できた範囲に限定する。

## 5. DB バックアップ要件

### 5.1 バックアップ対象

少なくとも以下をバックアップ対象とする。

- `numbers.db`
- 設定ファイル
- バージョン情報
- 運用に必要なドキュメント

### 5.2 バックアップタイミング

以下のタイミングでバックアップを検討する。

- 日次
- バージョンアップ前
- DB マイグレーション前
- ログ削除前
- 障害対応前

### 5.3 バックアップ方式

初期段階では、複雑な外部基盤を前提にせず、DB ファイルのコピーを基本とする。

ただし、コピー中の DB 整合性、保存場所、世代管理、復旧手順を明確にする。

### 5.4 世代管理

バックアップは単一世代ではなく、一定数の世代を保持する。

保持世代数は、容量、運用負荷、復旧要件を踏まえて決定する。

## 6. 復旧要件

### 6.1 復旧単位

障害時には、少なくとも以下を復旧できることを目標とする。

- DB ファイル
- 発券状態
- 処理状態
- 必要なログ
- 起動手順

### 6.2 復旧手順

復旧手順は、非技術者でも確認できるように文書化する。

最低限、以下を記載する。

- どのファイルを戻すか
- どこへ戻すか
- Docker をどの順序で停止・起動するか
- 復旧後に何を確認するか
- 復旧できない場合の連絡・判断手順

### 6.3 復旧確認

バックアップは取得するだけでなく、復旧できることを定期的に確認する。

## 7. ログ保持要件

### 7.1 ログの目的

ログは単なる容量削減の対象ではなく、以下の目的で利用される可能性がある。

- 障害解析
- 問い合わせ対応
- 業務状況の確認
- 操作・状態遷移の確認
- 監査・説明責任
- セキュリティ上の証跡

### 7.2 保持対象

以下のログについて、保持要否を整理する。

- 発券ログ
- 呼び出しログ
- 処理開始ログ
- 完了ログ
- 削除・取消に相当するログ
- エラーログ
- システム起動・停止に関するログ

### 7.3 保持期間

ログ保持期間は、技術都合だけではなく、業務上の必要性から決める。

例として、以下の区分を検討する。

- 当日分のみ必要なログ
- 数日から数週間必要なログ
- 月次確認まで必要なログ
- 障害解析・監査のため一定期間必要なログ

### 7.4 個人情報・機微情報

ログに個人情報、端末情報、操作者情報、IP アドレスなどが含まれる場合は、保持期間だけでなく、アクセス制御、マスキング、削除基準も検討する。

## 8. ログ削除・退避要件

### 8.1 削除前提にしない

ログ削除は、保持方針とバックアップ方針が決まった後に検討する。

容量対策だけを理由に、業務上必要なログを自動削除しない。

### 8.2 削除前退避

削除が必要な場合は、削除前に以下を検討する。

- DB バックアップ
- CSV またはテキスト形式での退避
- 日付単位でのアーカイブ
- 退避先の保存場所
- 退避ファイルのアクセス権限
- 退避ファイルの保持期間

### 8.3 自動削除

自動削除を導入する場合は、以下を明示する。

- 削除対象テーブル
- 削除条件
- 保持期間
- 実行タイミング
- 実行責任者
- 失敗時の扱い
- 実行ログ
- 削除前バックアップの有無

### 8.4 SQLite の注意点

SQLite では、DELETE のみでは DB ファイルサイズが縮小しない場合がある。

容量削減を目的とする場合は、VACUUM の要否を検討する。

ただし、VACUUM は DB ロックや処理時間に影響する可能性があるため、営業時間外実行、バックアップ取得、失敗時対応を含めて検討する。

## 9. 運用責任要件

### 9.1 判断責任

以下の変更は、単なる実装判断ではなく、運用判断を伴う。

- DB 保存場所の変更
- DB バックアップ方式の変更
- ログ保持期間の変更
- ログ削除機能の追加
- 自動削除処理の追加
- 外部ログ保存サービスの利用
- 復旧手順の変更

### 9.2 メンテナー判断と庁内判断

メンテナー判断で扱える変更と、庁内確認が必要な変更を分ける。

庁内確認が必要になりうる変更の例は以下とする。

- 業務記録の保持期間に影響する変更
- 障害時復旧方法に影響する変更
- 個人情報・機微情報の扱いに影響する変更
- 本番運用手順を変える変更
- 外部サービス利用を前提にする変更
- メジャーバージョンアップ相当の構成変更

## 10. Pull Request 受入条件

安定運用 Phase 2 に関する Pull Request では、以下を明記する。

- 変更の目的
- 現場業務上の利益
- 既存運用への影響
- DB・ログへの影響
- バックアップ要否
- 復旧手順への影響
- 非技術者メンテナーが確認すべき点
- 戻し方
- 動作確認内容
- AI を利用した場合の利用範囲

## 11. 初期対応案

Phase 2 の初期対応として、以下を優先する。

1. DB バックアップ方針の整理
2. 復旧手順の文書化
3. ログ保持方針の整理
4. ログ削除・退避方針の整理
5. 必要最小限のバックアップスクリプト検討
6. 必要最小限のログ退避・削除処理検討
7. 運用ドキュメント更新

## 12. 保留事項

以下は、現時点では要確認とする。

- 実際の本番運用環境
- DB 容量の増加速度
- 一日の発券件数
- ログの業務上の参照頻度
- 監査・問い合わせ対応で必要な保持期間
- 個人情報・機微情報の有無
- 庁内での情報資産管理ルール
- バックアップ保存先
- 障害時の責任分界

## 13. まとめ

#17 のログ定期削除は、単独で実装するより、#14 の DB 永続化・バックアップとあわせて、安定運用 Phase 2 の一部として扱うのが望ましい。

まずは、ログを消すかどうかではなく、DB をどう守り、どの情報をどの期間残し、障害時にどのように復旧するかを整理する。

そのうえで、必要に応じて、バックアップ、ログ退避、定期削除を段階的に実装する。
