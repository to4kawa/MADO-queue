# MADO-queue baseline audit task

## Goal

Establish a reproducible baseline for the current fork without changing production behavior.

## Required work

1. Run `python -m unittest test_app -v` in a clean environment.
2. Build and start the application with Docker Compose.
3. Verify the three HTML routes and `/display_data` respond.
4. Read all templates and JavaScript to map the actual request payloads used by the UI.
5. Compare actual API contracts with `docs/ARCHITECTURE.md`.
6. Add isolated tests for the cases below.
7. Record results in `docs/audit/BASELINE.md`.

## Test cases to add

### Request handling

- POST endpoints with no body.
- POST endpoints with malformed JSON.
- Unknown category for `/get_next_number`.
- Invalid and missing identifiers for processing endpoints.

### Ticket lifecycle

- Issue → start → complete.
- Issue → start → cancel → start again.
- Issue → delete.
- Start processing for a ticket that was never issued.
- Start the same issued ticket twice.
- Complete or cancel when more than one matching processing row exists.

### Number allocation

- First issue after daily reset for every category.
- Consecutive issues.
- Concurrent issues do not receive the same number.
- Behavior when category A reaches 499.
- Behavior when category B reaches 799.

### Printing

- Successful DB commit followed by printer exception.
- API response and persisted state after printer exception.
- Category D does not invoke printing.

### Database and migration

- Fresh schema generated by `init_db.py`.
- Representative legacy schema upgraded by `safe_migrate_db.py`.
- Compare column nullability, defaults, indexes, foreign keys and constraints.
- Run migration twice to verify idempotence.

### Time handling

- Timestamp omitted from issuance.
- Aware and naive ISO timestamps in processing.
- Date-boundary behavior under the documented JST assumption.

## Reporting rules

For each finding include:

- Classification
- Evidence
- Reproduction command or test name
- Expected behavior source, if any
- Actual behavior
- Operational impact
- Whether municipal workflow confirmation is required

Do not fix confirmed findings in this branch. Proposed fixes must be split into separate branches after review.
