# MADO-queue v1.0.0 Baseline Documentation Note

## 1. Purpose

This document records `v1.0.0` as the current baseline of MADO-queue for external audit, issue triage, and future change-management discussion.

This is not an upstream policy proposal. It is a fork-side audit note for understanding what `v1.0.0` already states, what it implies for operational stability, and how future issues or pull requests should be evaluated against the current baseline.

## 2. Baseline Position

`v1.0.0` should be treated as the first formal reference point for MADO-queue.

The baseline is not only a code snapshot. It also includes the documented operational assumptions, system scope, package position, and municipal use context.

At this baseline, MADO-queue is documented as:

- the first public package of MADO, named `queue`
- a ticket issuing and queue management system
- currently operating in the counter service workflow of Memuro Town
- designed as a lightweight package for small municipalities
- independently operated on a reception network
- not requiring network connection to internal municipal systems
- not designed to hold resident personal information

## 4. Product Scope at v1.0.0

### 4.1 Package Scope

The `queue` package is the first public package of the broader MADO system.

Other packages such as `hub`, `form`, `care`, and `move` are described as separate future repositories or future public packages. Therefore, changes to `queue` should not assume tight coupling with these future packages unless explicitly discussed.

### 4.2 Functional Scope

At `v1.0.0`, the system scope is limited to the following screens and responsibilities.

| URL | Purpose | Main user |
|---|---|---|
| `/` | Ticket issuing screen | Visitor / tablet |
| `/processing` | Processing management screen | Counter staff |
| `/display` | Public display screen | Waiting-room monitor |

The package supports small-start operation with the ticket issuing screen and receipt printer, while `/processing` and `/display` support operational visibility and call management.

### 4.3 Business Purpose

The documented purpose is not only faster processing. The system is intended to improve the visitor experience and create room for more careful counter service.

This matters for future changes: performance or developer convenience alone should not override the documented service model.

## 5. Operational Scope at v1.0.0

### 5.1 Standard Startup Model

The documented recommended startup method is Docker Compose.

```bash
docker compose up --build
```

The documented local startup path is:

```bash
pip install -r requirements.txt
python init_db.py
python app.py
waitress-serve --host=0.0.0.0 --port=8000 app:app
```

The baseline architecture also documents Waitress with `app:app` as the production WSGI target.

### 5.2 Data Location

The README documents that data is stored in:

```text
data/numbers.db
```

The README also states that backup can be performed by copying this file.

This statement should be treated as an operational assumption of `v1.0.0`, not as a complete backup policy.

### 5.3 Network and Security Assumptions

The baseline assumes:

- operation on a reception network
- no required connection to internal municipal systems
- no resident personal information stored by `queue`
- separation between visitor-facing and staff-facing screens by URL

These assumptions are part of the baseline and should be checked when future changes affect network topology, data storage, authentication, logging, or integration.

## 6. Data Model at v1.0.0

### 6.1 Main Tables

The architecture document defines the following main tables.

| Table | Purpose |
|---|---|
| `numbers` | current category counters |
| `event_logs` | issued ticket records |
| `processing_logs` | ticket processing lifecycle records |

### 6.2 `numbers`

`numbers` manages current counters by category.

Baseline categories:

| Category | Initial value | Purpose |
|---|---:|---|
| A | 1 | general / shorter procedures |
| B | 500 | registration / transfer procedures |
| C | 800 | special procedures |
| D | 0 | non-printing visitor count |

### 6.3 `event_logs`

`event_logs` records issued tickets.

Key fields include:

- `id`
- `category`
- `button_text`
- `timestamp`
- `current_number`
- `staff_count`

### 6.4 `processing_logs`

`processing_logs` records the processing lifecycle.

Key fields include:

- `ticket_number`
- `category`
- `button_text`
- `start_time`
- `end_time`
- `wait_time`
- `status`
- `processing_time`
- `created_at`
- `staff_count`
- `event_log_id`

The presence of these log tables means that log handling should not be treated only as a storage cleanup problem. It may affect inquiry response, operational review, auditability, troubleshooting, and future analysis.

## 7. Requirements at v1.0.0

### 7.1 Functional Requirements

The requirements document includes the following major functional areas.

- ticket issuing
- category management
- call / processing management
- public display
- data recording and analysis
- deletion / cancellation handling

Of particular importance:

- tickets are reset daily at midnight
- ticket issue logs are recorded
- processing logs are recorded
- deleted tickets are retained as history with `deleted` status

### 7.2 Non-functional Requirements

The requirements document includes the following non-functional areas.

- response and update intervals
- availability during opening hours
- tablet / PC / large monitor usability
- receipt printer operation
- reception-network independent operation
- no resident personal information storage
- SQLite single-node operation
- Docker as the standard startup method
- Windows local execution support

## 8. Baseline Interpretation for Change Management

### 8.1 What Should Be Treated as Baseline-Preserving

The following changes are likely to preserve the `v1.0.0` baseline if they do not change documented behavior or operational assumptions.

- documentation corrections
- startup instruction clarification
- test additions that do not change behavior
- small bug fixes that preserve existing API and DB schema
- explanatory notes for existing Docker or local startup procedures
- clarification of already documented DB backup assumptions

### 8.2 What Should Require Additional Review

The following changes should require additional review because they may affect the documented baseline.

- DB schema changes
- data retention changes
- log deletion or automatic purge behavior
- startup target changes, such as changing `app:app`
- project structure changes that alter documented commands
- Python version requirement changes
- dependency locking or dependency major changes
- Docker Compose behavior changes
- changes to where `numbers.db` is stored
- changes that affect Windows local execution
- changes that affect reception-network independence
- changes that introduce external services or external DBs

### 8.3 What May Be Major-Version-Level Changes

The following may be major-version-level changes because they affect the way municipalities, maintainers, or external contributors understand and operate the package.

- moving the application into a new package layout if documented commands change
- changing configuration file format
- changing production startup commands
- changing the supported Python baseline
- changing the persistence model
- changing the backup and restore model
- changing log retention or deletion behavior
- introducing mandatory external infrastructure
- changing assumptions about personal information, network isolation, or integration

These changes may still be valuable, but they should be described as baseline-changing proposals rather than routine developer-experience improvements.

## 10. Practical Use

This document can be used when reviewing future issues and pull requests.

Reviewers can ask:

1. Does the proposal preserve the `v1.0.0` baseline?
2. Does it change documented startup, DB, logging, or operational assumptions?
3. Can a non-engineer maintainer understand the operational effect?
4. Does the proposal require municipal explanation or internal review?
5. Is the change small enough to merge as a maintenance change, or should it be treated as a larger versioned change?

## 11. Current Assessment

`v1.0.0` already provides a useful public baseline, but it is not a complete operations manual.

The next safe step is not to immediately expand implementation scope. The next safe step is to document the boundary between:

- baseline-preserving maintenance
- stable-operation Phase 2 requirements
- major-version-level structural changes
- municipal governance or internal decision topics

This separation allows external contributors to help without accidentally shifting municipal operational responsibilities beyond what maintainers can safely review.
