# repo 文書化プロセス

この文書は、MADO-queue を文書化するために実際に読んだ順番と、他の同種プロジェクトにも転用できる文書化手順をまとめたものです。目的は、単発の README 整備ではなく、属人化を減らすための再利用可能なプロセスを残すことです。

## この repo をどう読んだか

今回の文書化では、推測ではなくリポジトリ内の事実を優先しました。特に次の情報源を確認しました。

1. 作業前の Git 状態とブランチ。
2. `AGENTS.md` の作業制約。
3. 既存ドキュメント: `docs/REQUIREMENTS.md`, `docs/ARCHITECTURE.md`, `docs/DEVELOPMENT.md`。
4. 実装の入口: `app.py`。
5. DB 関連: `init_db.py`, `safe_migrate_db.py`, `config.py`。
6. 起動設定: `Dockerfile`, `docker-compose.yml`, `entrypoint.sh`, `requirements.txt`。
7. UI: `templates/index.html`, `templates/syori.html`, `templates/display.html`, `static/script.js`, `static/style.css`。
8. 既存 README と docs/audit 配下の監査文脈。

## どの順番で確認したか

### 1. 作業安全性の確認

- 現在ブランチを確認しました。
- 作業ツリーの未コミット変更を確認しました。
- 指定ブランチ `docs/repo-documentation-process` を作成して切り替えました。

### 2. 既存の意図を確認

- `AGENTS.md` から、現段階では本番挙動の変更ではなく、監査・証拠収集・文書化を優先する方針を確認しました。
- `docs/REQUIREMENTS.md` から、業務目的、対象ユーザー、カテゴリ、非機能要件を確認しました。
- `docs/ARCHITECTURE.md` から、既存の API、DB、画面、印刷仕様を確認しました。
- `docs/DEVELOPMENT.md` から、開発・テスト・Docker 起動手順を確認しました。

### 3. 実装から事実を確認

- `app.py` でルート、API、DB 操作、印刷処理、入力検証を確認しました。
- `init_db.py` と `safe_migrate_db.py` で新規 DB と移行 DB の責務を確認しました。
- `config.py` でカテゴリ開始番号を確認しました。
- Docker 関連ファイルで起動方法、DB 永続化、依存 OS パッケージを確認しました。
- テンプレートと JavaScript で、画面操作と API 呼び出しのつながりを確認しました。

### 4. 文書へ落とし込んだ観点

- 初見の開発者が最初に読む README。
- システム全体像を掴む architecture。
- 日常運用を支える operation。
- 他 repo へ横展開できる documentation-process。

## 他 repo にも使える文書化手順

### Step 0: 作業前の安全確認

- 現在ブランチを確認する。
- 未コミット変更の有無を確認する。
- 指定があれば専用ブランチを切る。
- 既存の作業者変更を上書きしない。

### Step 1: 既存ドキュメントを読む

読む対象の例:

- README
- docs 配下
- CONTRIBUTING
- SECURITY
- CODE_OF_CONDUCT
- AGENTS.md や開発者向け指示ファイル
- 既存の設計メモ、監査メモ、運用メモ

見る観点:

- この repo の目的。
- 対象ユーザー。
- サポートする実行方法。
- 既知の制約。
- 変更してよい範囲と、してはいけない範囲。

### Step 2: 実装の入口を特定する

代表例:

- Web アプリ: `app.py`, `main.py`, `server.ts`, `src/main.*`
- CLI: `__main__.py`, `cmd/`, `bin/`
- フロントエンド: `package.json`, `src/`, `pages/`, `app/`
- インフラ: `Dockerfile`, `docker-compose.yml`, `.github/workflows/`

確認すること:

- 起動コマンド。
- 主要ルート / コマンド。
- 外部依存。
- 設定値と環境変数。
- 永続化データの置き場所。

### Step 3: データの流れを追う

- 入力元は何か。
- どの API / 関数を通るか。
- どこに保存されるか。
- どの画面 / 出力 / 外部サービスに出るか。
- エラー時に何が残るか。

### Step 4: 操作の流れを追う

- 初回セットアップ。
- 通常起動。
- よく使う画面やコマンド。
- 停止。
- リセット / 復旧。
- テスト。

### Step 5: 「事実」と「推測」を分ける

文書では次を分けて書くと、属人化や誤解を減らせます。

- コードから確認できること。
- 設定ファイルから確認できること。
- 既存ドキュメントに書かれていること。
- 実運用を見ないと分からないこと。
- ドメイン担当者に確認が必要なこと。

### Step 6: 文書を役割別に分ける

| 文書 | 主な読者 | 書くべきこと |
| --- | --- | --- |
| README | 初見の開発者、導入検討者 | 概要、主要機能、セットアップ、構成、注意点。 |
| architecture | 開発者、レビュー担当 | 全体構成、処理フロー、コンポーネント、データ・設定の流れ。 |
| operation | 運用担当、将来の保守者 | 起動停止、初回設定、コマンド、トラブルシュート、確認箇所。 |
| documentation-process | 横展開する人、将来の自分 | どう読んだか、再利用可能な手順、チェックリスト、未判断事項。 |

## 再利用可能なチェックリスト

### リポジトリ概要

- [ ] この repo は何を解決するものか。
- [ ] 誰が使うものか。
- [ ] 主要な機能は何か。
- [ ] 本番利用中か、検証用か、ライブラリか。
- [ ] ライセンスは何か。

### セットアップ・実行

- [ ] 推奨起動方法は何か。
- [ ] 代替起動方法はあるか。
- [ ] 必要な言語 / ランタイム / OS パッケージは何か。
- [ ] 依存関係のインストール方法は何か。
- [ ] 初回だけ必要な作業は何か。
- [ ] 停止方法は何か。

### 設定・環境変数

- [ ] 必須環境変数は何か。
- [ ] 任意環境変数は何か。
- [ ] デフォルト値はどこで決まるか。
- [ ] 秘密情報があるか。
- [ ] 環境ごとの差分は何か。

### データ・永続化

- [ ] DB やファイルはどこに保存されるか。
- [ ] 初期化方法は何か。
- [ ] 移行方法は何か。
- [ ] バックアップ対象は何か。
- [ ] 消してよいファイルと消してはいけないファイルは何か。

### API / 画面 / ジョブ

- [ ] 主要 URL またはコマンドは何か。
- [ ] 入力と出力は何か。
- [ ] 自動実行される処理はあるか。
- [ ] ポーリング、スケジューラー、キューなどはあるか。
- [ ] エラー時の戻り値や副作用は何か。

### 外部依存

- [ ] 外部サービスに接続するか。
- [ ] ハードウェア依存があるか。
- [ ] ネットワーク前提があるか。
- [ ] ブラウザや端末の前提があるか。
- [ ] OS 固有の依存があるか。

### テスト・確認

- [ ] 既存テストの実行コマンドは何か。
- [ ] テストは外部サービスや実機を必要とするか。
- [ ] 一時 DB やスタブを使う方法はあるか。
- [ ] 手動確認コマンドは何か。
- [ ] CI があるか。

### 運用

- [ ] 開始・終了手順は何か。
- [ ] ログはどこで見るか。
- [ ] 生成物はどこにあるか。
- [ ] 障害時の切り分け手順は何か。
- [ ] バックアップ / 復旧手順はあるか。

### 未確認事項

- [ ] コードからは判断できない運用ルールを列挙したか。
- [ ] ドメイン担当者への確認事項を列挙したか。
- [ ] 推測を断定として書いていないか。
- [ ] 既存ドキュメントと実装の差分を明記したか。

## 今回判断できなかった点

- 実運用環境の正確な OS、プリンター機種、USB 権限、ドライバー設定。
- 本番 DB のバックアップ頻度、保存期間、復旧演習の有無。
- カテゴリ番号帯の上限をシステムで強制すべきか、運用上の目安なのか。
- 認証なしで職員処理画面を提供する前提が、どの導入環境でも妥当か。
- カテゴリ・ボタン名の変更を設定化する必要があるか、フォークごとのテンプレート編集で十分か。
- 既存の uppercase `docs/ARCHITECTURE.md` と今回追加した lowercase `docs/architecture.md` の今後の統合方針。

## 文書化プロセスとしての学び

- 既存 README をいきなり書き換える前に、業務要件・実装・運用手順を分けて読むと、断定してよい範囲が明確になります。
- 小規模業務アプリでは、コードよりも「どの端末で、誰が、いつ、何を見るか」が属人化しやすいため、operation 文書が重要です。
- 実機依存がある repo では、テスト可能な範囲と実機確認が必要な範囲を明示すると、開発者が安全に参加できます。
- 新規 DB 作成スクリプトと移行スクリプトが分かれている場合、両方を文書化対象にする必要があります。
- 「不明」「要確認」を残すことは文書の未完成ではなく、推測による誤運用を避けるための重要な情報です。
