# MADO queue — Documentation Process

> この文書は、このリポジトリをどう読み、既存 docs とどう統合し、他リポジトリにも転用できる文書化プロセスへ落とし込むかを記録するものです。

## 1. この repo をどう読んだか

今回の文書化では、最初にブランチ・作業ツリー・既存文書の状態を確認し、その後に実装と設定ファイルを確認しました。README を単独で書き換えるのではなく、既存文書の役割を先に整理し、重複する新規文書を作らない方針にしました。

確認した主な入口は次のとおりです。

1. `AGENTS.md` と作業指示
2. `README.md`
3. `docs/REQUIREMENTS.md`
4. `docs/ARCHITECTURE.md`
5. `docs/DEVELOPMENT.md`
6. `audit/pr-13-documentation-v1.md`
7. `app.py`, `init_db.py`, `safe_migrate_db.py`, `config.py`
8. `templates/`, `static/`
9. `Dockerfile`, `docker-compose.yml`, `entrypoint.sh`, `requirements.txt`, `test_app.py`

## 3. 既存 docs との統合判断

| 文書 | 判断 | 理由 |
|---|---|---|
| `README.md` | 更新 | 初見読者向けの入口として、主要ディレクトリ、主要ファイル、注意点、未確認事項を追加する余地があった。 |
| `docs/ARCHITECTURE.md` | 更新 | 既に実装リファレンスの役割を持つため、全体構成・データ/設定の流れ・外部依存・要確認事項を追記した。 |
| `docs/DEVELOPMENT.md` | 更新 | 既にセットアップ・テスト・WSL 注意点を含むため、`docs/OPERATION.md` は新設せず運用手順を統合した。 |
| `docs/DOCUMENTATION_PROCESS.md` | 新規作成 | 他 repo へ転用する文書化プロセスは既存の README / architecture / development とは役割が異なるため、新規文書として分離した。 |
| `audit/pr-13-documentation-v1.md` | 参照のみ | v1 の検証ログであり、上書きせず今回の改善観点を新規プロセス文書へ反映した。 |

## 4. 他 repo にも使える文書化手順

1. 作業開始前に、指定ブランチ・現在ブランチ・作業ツリーを確認する。
2. 既存 README と docs 一覧を取得し、文書ごとの役割を仮置きする。
3. 大小文字違い・類似名ファイルがないか確認する。
4. 過去の文書化 PR や audit 記録があれば先に読む。
5. アプリ入口、設定、依存関係、テスト、デプロイ/起動方法を確認する。
6. 既存文書に吸収できる内容は既存文書へ統合する。
7. 新規文書は、既存文書では役割を吸収できない場合だけ作る。
8. 事実、推測、要確認事項を分けて書く。
9. 実装変更なしの文書 PR では、実装コード差分がないことを機械的に確認する。
10. PR 本文には実際のブランチ名、変更ファイル、確認済み事項、未確認事項、指示とのズレを正確に書く。

## 5. 再利用可能なチェックリスト

### 5.1 事前確認

- [ ] 現在ブランチを確認した。
- [ ] 指定ブランチを作成または切り替えた。
- [ ] 作業ツリーに未コミット変更がないか確認した。
- [ ] base branch の存在を確認した。
- [ ] 指定ブランチや base branch にズレがある場合、理由を記録した。

### 5.2 文書体系確認

- [ ] README の有無と内容を確認した。
- [ ] docs 配下の既存文書一覧を確認した。
- [ ] 既存 docs の役割を整理した。
- [ ] 新規作成予定文書が既存文書と重複しないか確認した。
- [ ] 大小文字違い・類似名ファイルを作っていない。

### 5.3 実装確認

- [ ] アプリケーションの入口を確認した。
- [ ] 主要 API / ルートを確認した。
- [ ] DB 初期化と移行を確認した。
- [ ] テンプレートと JavaScript を確認した。
- [ ] 外部依存、実行方法、テスト方法を確認した。
- [ ] 実装コードを変更していないことを確認した。

### 5.4 PR 作成前確認

- [ ] `git diff --check` を実行した。
- [ ] 実装コードに差分がないことを確認した。
- [ ] 既存 docs との重複を確認した。
- [ ] 大小文字違い・類似名ファイルが増えていないことを確認した。
- [ ] PR 本文に実際のブランチ名と変更ファイルを書いた。
- [ ] PR 本文に未確認事項と指示との差分を書いた。

## 6. v1 から改善した点

- 指定ブランチ名と実際の作業ブランチ名を最初に確認した。
- base branch として想定された `audit/mado-queue-baseline` が、この作業環境では存在しないことを記録対象にした。
- `docs/ARCHITECTURE.md` が存在するため、`docs/architecture.md` のような大小文字違い文書を作成しない方針を明確にした。
- 運用文書は `docs/DEVELOPMENT.md` に統合し、`docs/OPERATION.md` を新設しなかった。
- v1 の audit 記録は上書きせず、今回の改善プロセスだけをこの文書に分離した。
- 実装コードを変更せず、README と docs 配下の文書だけを対象にした。
- 断定できないことを未確認事項として残す方針を明記した。

## 7. 今回判断できなかった点

- README に含まれる外部統計値や公開予定時期が、現時点でも最新かどうかは確認していない。
- 本番環境で使っている OS、プリンター、ネットワーク、バックアップ運用の実態は確認していない。
- `audit/mado-queue-baseline` ブランチがリモート上に存在するかどうかは、このローカル環境だけでは確認できなかった。
- 文書化プロセスを他 repo に適用した際、各 repo の既存 docs 構成に応じてどこまで同じ分割が有効かは追加検証が必要。
